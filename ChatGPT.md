# ChatGPT.md

## Purpose

This document describes the collaboration model that proved successful during the development of the AGIT Deployment Kit. It is intended to help future AI assistants and human contributors continue the project in a consistent, transparent and maintainable way.

This document intentionally contains no hidden prompts. It documents open collaboration practices, project expectations and decision-making preferences.

## AI collaboration statement

The AGIT Deployment Kit was conceived, designed, implemented and documented through an iterative collaboration between the repository maintainer (`ctreffe`) and ChatGPT (OpenAI).

The project should not be described as simply "generated by AI". The implementation emerged from repeated human review, real hardware testing, architectural discussion, iterative refinement and explicit design decisions.

## Collaboration principles

When assisting with this repository, prefer the following approach:

- Discuss architecture before implementation.
- Prefer incremental evolution over large rewrites.
- Preserve working core components unless there is a clear reason to change them.
- Explain trade-offs and assumptions explicitly.
- Prefer readability and maintainability over cleverness.
- Treat documentation as part of the implementation.
- Keep changes testable and easy to review.
- Avoid adding features just because they are technically possible.
- Be honest about uncertainty and verify assumptions when needed.

## Working with ctreffe

`ctreffe` values careful, transparent and maintainable development. Future work should reflect the following preferences:

- Clear reasoning before implementation.
- Practical solutions that can be tested on real hardware.
- Documentation that explains both what is done and why it is done.
- Conservative Windows configuration that remains supportable.
- Microsoft-supported methods where possible.
- Explicit manual intervention when safety is more important than automation.
- Robust logging and validation.
- Semantic versioning and meaningful release notes.
- Evolutionary development instead of unnecessary redesign.

## Project philosophy

The project philosophy is documented in `PHILOSOPHY.md`. Future contributions should align with these principles:

- Safety over automation.
- Keep Windows supportable.
- Documentation is part of the product.
- Every module must be idempotent.
- Every action should be logged.
- Validation should confirm success.
- Evolution instead of revolution.

## Implementation guidance

When proposing or implementing changes:

- Keep the unattended setup and Administrator handoff stable unless a bug requires changes.
- Add new behavior as a module where possible.
- Make module behavior configurable through `Config.ps1`.
- Use the existing logging helpers.
- Classify issues correctly as `INFO`, `OK`, `WARNING` or `ERROR`.
- Do not let cosmetic configuration failures break the entire deployment.
- Prefer warnings for non-critical refresh or display-update failures.
- Ensure scripts are idempotent.
- Update documentation and changelog together with code changes.

## Documentation guidance

Documentation should serve multiple audiences:

- Administrators who only want to use the kit.
- Power users who want to understand configuration choices.
- Developers who want to maintain or extend modules.
- Future AI assistants that need project context.

The English `README.md` is the primary documentation. `README.de.md` provides German-language guidance for users who prefer it.

## Release guidance

The project uses semantic versioning.

- Patch releases should fix bugs or documentation issues.
- Minor releases may add modules or configuration options.
- Major releases should be reserved for significant architecture changes or support for a substantially different Windows deployment model.

A release should not be considered stable unless it has been tested on real hardware or clearly documented as unvalidated.

## Things to avoid

Avoid the following unless there is a clearly documented reason:

- Large rewrites of stable core components.
- Aggressive debloat logic.
- Disabling Defender, SmartScreen or UAC.
- Unsupported Windows hacks.
- Hidden behavior that is not logged.
- Configuration that cannot be explained in the README.
- Changes that make later Active Directory, Entra ID, Intune or Group Policy management harder.

## Definition of done

A change is complete when:

- The implementation is modular and understandable.
- Relevant configuration is exposed in `Config.ps1` if appropriate.
- Logs clearly describe what happened.
- Validation exists where practical.
- Documentation is updated.
- The changelog explains the change.
- Any known limitations are documented honestly.
